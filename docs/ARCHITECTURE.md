# Architecture

## 总体架构

系统采用清晰的单体应用优先架构，先完成真实数据驱动的 MVP 闭环，再考虑复杂架构拆分。第一版默认由 FastAPI 后端、Vue 3 + Element Plus 前端、PostgreSQL 数据库和定时任务组成。

## 部署资源约束

- 个人云服务器部署时，PostgreSQL 必须限制内存和连接数，避免与同机其他项目共同运行时耗尽 2c2g 服务器资源。
- 推荐在 PostgreSQL 启动配置中显式设置较小的 `shared_buffers`、`work_mem`、`maintenance_work_mem` 和 `max_connections`，并按实际服务器内存继续收紧。
- 本项目日常只应执行自动交易辅助所需任务：首次历史数据初始化只运行一次，之后每天盘后拉取当天数据并生成交易计划；不要在同一台 2c2g 服务器上长期运行开发测试、批量补数或重计算任务。

## 模块边界

- 数据采集模块：从 TuShare、AkShare、东方财富或腾讯行情拉取市场、指数、板块、个股、涨跌停和资金流数据。
- 市场环境模块：计算市场评分和市场状态，生成总仓位建议。
- 强势板块模块：计算板块涨幅、持续性、成交额变化、涨停数量、强势股数量和板块评分。
- 候选筛选模块：执行趋势强势、放量突破、强势回踩三类策略，并完成基础过滤。
- 交易计划模块：把候选股票转成第二天条件交易计划，包含买入条件、止损价、止盈价、仓位和风险提示。
- 盘中跟踪模块：跟踪计划是否触发、取消或转为风险状态。
- 复盘模块：生成交易复盘记录，统计策略胜率、板块胜率和盈亏表现。
- 模拟交易模块：基于交易计划模拟买入、卖出、持仓、交易记录、资金曲线和风险指标。
- Web 模块：展示 P0 页面和后续 P1 页面，支持筛选、排序、导出和查看理由。

## 数据流

```text
数据源
  -> 本地 PostgreSQL
  -> 指标计算
  -> 市场环境 / 强势板块 / 候选筛选
  -> 交易计划
  -> 盘中跟踪 / 复盘 / 模拟交易
  -> Web 展示
```

## 核心数据表

MVP 起步至少包含以下业务表：

- `market_daily`：每日市场环境。
- `sector_daily`：每日强势板块。
- `trade_plan`：每日交易计划。
- `trade_review`：每日交易复盘。

模拟交易扩展至少包含：

- `simulation_account`：模拟账户。
- `simulation_position`：当前持仓。
- `simulation_trade`：模拟交易记录。
- `simulation_equity_curve`：资金曲线。

## 防未来函数

- 交易计划只能使用计划生成时点及以前的数据。
- 盘后计划生成不能读取目标交易日的盘中或收盘数据。
- 复盘统计可以使用事后数据，但不得反向改写当时的入选理由和买入条件。
- 回测、模拟交易和实盘跟踪必须共享同一套可解释交易规则，不能为了提高历史收益调整成交假设。

## 接口边界

MVP API 至少包含：

- `GET /api/market/today`
- `GET /api/sectors/strong?date=YYYY-MM-DD`
- `GET /api/trade-plans?date=YYYY-MM-DD`
- `GET /api/trade-plans/{id}`
- `PATCH /api/trade-plans/{id}/status`
- `GET /api/reviews?date=YYYY-MM-DD`
- `POST /api/reviews`
- `PATCH /api/reviews/{id}`

后续模拟交易 API 必须围绕账户、持仓、交易记录和资金曲线展开，不能绕过 `trade_plan` 直接交易计划外股票。

## 运行任务

- 盘后生成交易计划：交易日 16:30 以后运行。
- 盘中跟踪计划状态：交易日 09:30 - 15:00，每 1 - 5 分钟运行一次。
- 收盘复盘任务：交易日 15:30 - 16:30 运行。
- 数据源延迟或失败时，要保留可重跑入口和错误证据。
